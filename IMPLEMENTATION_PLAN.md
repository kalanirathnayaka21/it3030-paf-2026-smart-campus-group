# Smart Campus - Project Implementation Plan

This document outlines the recommended order of implementation for the Smart Campus project features, categorized by module.

---

## Phase 1: Foundation & Security (Module E)
**Goal:** Establish the security architecture and user identity before building business logic.
- [ ] **E.1 OAuth 2.0 Integration:** Implement Google Sign-In for authentication.
- [ ] **E.2 Role-Based Access Control (RBAC):** Define `USER`, `ADMIN`, and `TECHNICIAN` roles.
- [ ] **E.3 Secure API Endpoints:** Protect backend routes using JWT/OAuth tokens and secure frontend routes.

## Phase 2: Core Infrastructure (Module A)
**Goal:** Create the database and UI for the physical resources of the campus.
- [ ] **A.1 Resource Schema:** Define metadata for Lecture Halls, Labs, Meeting Rooms, and Equipment.
- [ ] **A.2 Admin CRUD:** Implement functionality for Admins to add/edit/delete resources (Status: ACTIVE/OUT_OF_SERVICE).
- [ ] **A.3 Search & Filtering:** Implement the catalogue view with filters for type, capacity, and location.

## Phase 3: Resource Booking (Module B)
**Goal:** Implement the primary resource utilization workflow.
- [ ] **B.1 Booking Requests:** Enable users to request resources with date, time, purpose, and attendees.
- [ ] **B.2 Conflict Prevention:** Implement backend logic to prevent overlapping bookings for the same resource.
- [ ] **B.3 Approval Workflow:** Build the Admin dashboard to review, approve, or reject (with reasons) requests.
- [ ] **B.4 User Booking View:** Allow users to view and cancel their own bookings.

## Phase 4: Maintenance & Ticketing (Module C)
**Goal:** Add support for reporting issues and managing repairs.
- [ ] **C.1 Incident Reporting:** Create the form for users to report issues with categories and priority.
- [ ] **C.2 Image Attachments:** Support up to 3 image uploads for evidence.
- [ ] **C.3 Technician Workflow:** Implement ticket assignment and status updates (OPEN → RESOLVED → CLOSED).
- [ ] **C.4 Communication:** Implement the comment system on tickets with ownership rules.

## Phase 5: User Engagement & Feedback (Module D)
**Goal:** Ensure users are informed of system actions.
- [ ] **D.1 Notification Engine:** Implement triggers for booking updates, ticket status changes, and new comments.
- [ ] **D.2 Web UI Notification Panel:** Create a real-time or polling-based notification center in the frontend.

---

### Implementation Rationale:
1.  **Module E First:** Security is difficult to "bolt on" later. Setting up OAuth 2.0 and roles early ensures every API endpoint you build is secure from the start.
2.  **Module A Second:** Modules B and C both depend on the existence of Resources (Lecture halls, projectors, etc.).
3.  **Module B Third:** Booking is the core utility of the platform.
4.  **Module C Fourth:** Ticketing adds maintenance capability once the catalogue is stable.
5.  **Module D Last:** Notifications act as a wrapper around the events generated by Modules B and C.
